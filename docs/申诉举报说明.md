# 小程序 · 申诉与举报

## 入口

| 位置 | 能力 |
|------|------|
| 订单详情 | **订单申诉**（进行中订单，买卖双方） |
| 商品详情 | **举报**（非发布者，底栏第二行） |
| 我的 → 我的申诉/举报 | 本人发起的记录；**待受理**可撤销 |
| 我的 → 收到的举报 | **被举报方**查看案卷并回应（无微信推送） |

## 通知说明

**无微信模板消息**；动态写入 Bmob **`UserNotice`** 表，在 **消息 Tab → 通知** 展示，底部 Tab 显示未读角标。

触发：收到举报、订单申诉、案卷回应、撤销、管理员受理/驳回/裁决。

也可从 **我的 → 收到的举报** 进入案卷。

### 管理员结案后收不到通知？

1. **无微信推送**：只能打开小程序 → **消息 → 通知** 查看（或看底部 Tab 角标）。
2. **云函数未更新**：Bmob 上的 `adminDisputeDecide` 须重新部署（粘贴 `admin-web/cloudfunctions/bundled/adminDisputeDecide.bmob.js` 全文）。旧版结案不会写 `UserNotice`。
3. **管理端已兜底**：本地管理端结案成功后会自动写 `UserNotice`；提示「已向 N 位用户发送站内通知」即成功。
4. **历史结案补发**：在 `admin-web` 执行  
   `node scripts/backfill-closed-notices.cjs <ApplicationId> <REST_API_KEY>`
5. **核对数据**：Bmob 控制台 → `UserNotice` 表应有 `type=case_closed` 记录，`userId` 为对应用户 objectId。

## 案卷归并（caseKey）

- 同一 **商品** 的多次举报：`caseKey = ITEM:{商品objectId}`，自动归入一案卷。
- 同一 **订单** 的申诉：`caseKey = ORDER:{订单objectId}`。
- 过程记录在 **DisputeReply** 表：举报方陈述、被举报方回应、参与方补充说明，管理端按案卷展开查看。

申诉/举报可上传最多 **3 张** 凭证图（云存储 `ticket/`，写入 `evidence` 字段 JSON）。

## 实现

- `utils/ticket.js`：写入 Bmob **`Dispute`** 表；订单申诉同时将 **`Order.frozen = true`**
- `pages/feedback/feedback`：通用提交/列表页

## 必做：Bmob 建表（当前报错原因）

若提交时出现 **`object not found for Dispute`**，说明库里**还没有 `Dispute` 表**（与截图左侧表列表一致）。

### 方式 A：控制台手动建表（推荐）

1. 登录 [Bmob 控制台](https://www.bmobapp.com/) → 你的应用 → **数据浏览**
2. **创建表**，表名必须为：`Dispute`（区分大小写）
3. **添加列**（类型选字符串 / 数字 / 布尔，与下表一致）：

| 列名 | 类型 | 说明 |
|------|------|------|
| ticketType | 字符串 | `DISPUTE` 或 `REPORT` |
| caseKey | 字符串 | 案卷键 `ITEM:{id}` / `ORDER:{id}` |
| caseTitle | 字符串 | 案卷标题摘要 |
| orderId | 字符串 | 订单申诉时填 Order.objectId |
| targetType | 字符串 | 举报时：`Item` 或 `_User` |
| targetId | 字符串 | 被举报对象 id |
| respondentId | 字符串 | 被举报方用户 id |
| submitterId | 字符串 | 提交人 _User.objectId |
| reason | 字符串 | 说明 |
| evidence | 字符串 | 图片 JSON 数组（cloud://） |
| status | 字符串 | `SUBMITTED` / `UNDER_REVIEW` / `CLOSED` |
| liabilityParty | 字符串 | 结案后：`none` / `buyer` / `seller` |
| orderOutcome | 字符串 | 结案后：`COMPLETED` / `CANCELLED` |
| creditDelta | 数字 | 可选 |
| decisionNote | 字符串 | 管理员说明 |

另需表 **`DisputeReply`**（过程链）、**`UserNotice`**（站内通知），字段见 `admin-web/docs/Bmob建表字段说明.md`。

4. 打开 **`Order`** 表 → **添加列** → `frozen`（**布尔 Boolean**，默认 false）

5. 保存后回到小程序重新编译，再试「提交申诉」

### 方式 B：脚本自动建 Dispute 等表

在 `admin-web` 目录执行（把 Application ID、REST API Key 换成你应用的）：

```bash
cd admin-web
node scripts/init-bmob-tables.cjs <ApplicationId> <REST_API_KEY>
```

脚本会插入一条占位数据从而创建 `Dispute` 表；**Order.frozen 仍需在控制台手动添加**。

建表后可在数据浏览删除占位行。

## 管理端

工单在 Web 后台 **申诉与举报** 菜单处理，详见 `admin-web/docs/申诉举报处理流程.md`。

需部署云函数：`adminDisputeList`、`adminDisputeDecide`。当前小程序仓库已提供可部署版本：

- `cloudfunctions/adminDisputeDecide/index.js`：裁决后关闭案卷、调整责任方信用分、推进订单终态、解冻订单并写站内通知。
- `cloudfunctions/creditSweep/index.js`：建议配置为每 5 分钟定时任务，兜底处理付款锁定超时和长期待确认订单。

**运营流程**：筛选「已提交」→ 受理 → 裁决（或驳回）→ 用户在小程序 **消息 → 通知** 查看结案（无微信推送）。

项目总说明：[../../README.md](../../README.md) 第五节。

## 可选云函数

`cloudfunctions/submitDispute`、`submitReport` 可与 SDK 直写二选一，非必须。
